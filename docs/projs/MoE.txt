根据 Dense LLM accounting 的结果, 线性运算占总运算的 90% 以上, 以线性运算计, 前向运算量 FLOPs = 2*参数量，反向运算量 FLOPs = 4*参数量
也就是说, 训练运算量 FLOPs-train = 6*参数量, 推理运算量 FLOPs-infer = 2*参数量
在经典的decoder架构中, attention部分参数量 P_attn : ffn部分参数量 P_ffn = 1:2，也就是说 FLOPs-ffn = 2/3 * FLOPs

MoE LLM 就是把 大的FFN 部分改成多个 expert-FFN 网络, 每次运算只激活其中 top-k 个 export-FFN. 考虑 N 个专家, 每次激活 k 个.
Params-active = Params-attn + k * Params-expert
Params-total = Params-attn + N * Params-export

在设计 MoE 模型时, 如果参考 经典decoder架构, Params-attn = 1/2 * k * Params-export, 即每次运算中, attn部分参数量 = 1/2 * ffn部分参数量
故 active-ratio = (P_attn + k*P_expert) / (P_attn + N*P_expert) = (k/2 + k) / (k/2 + N) = 3k/(2N+k)

---> 可以保持FLOPs的同时，增加exports数量, 以获得更多专业知识储存 ---> 相同激活参数量下，MoE 一般具备更好的效果
---> 相同激活参数下（比如1.3B dense模型 vs 1.3B/6.9B MoE模型, 达到相同的效果（training loss/validation loss/HellaSwag等）所需的tokens/training time，
     MoE模型只需dense模型的 1/3 - 1/2


MoE模型可以实现独特的 token粒度-专家匹配，并在FFN部分作 数据流并行化 ---> Expert-Parallelism
首先LLM pretrain必备DP模式：DP即 Data Parallelism，即 不同的 batch数据 交给 完全相同的model-optimizer副本，计算出不同的grad, all-reduce并执行相同的更新
其次, MoE LLM 共享 attention 部分, 但是却将 FFN部分的不同专家并行部署在不同设备上.

-->当每个设备完成各自的 attention 之后, 各自得到了不同batch的 tokens 激活值 [B, S, D]. 然后通过一个路由网络 router, 计算各自每个 token 应该去哪个设备
   然后是 all-to-all dispatch: 即每台设备留下属于自己的tokens, 发送其他tokens去该去的设备，并接收来自其他设备、该自己计算的tokens，组装成 [N, D]数据
   执行完 expert-parallel-FFN 之后是 all-to-all combine: 即将接收的tokens发送回原来的位置
(Typically models are trained using data-parallelism where different machines will get different slices of the trainig/inference data.
 The machines used for operating on the different slices of data can now be used to host many more model parameters)
-->DP必然多机，多机必然总是可以host更多专家参数，而EP解决了专家并行的问题，从而吞吐量更高.

--> MoE模型的效果一般更好：以Qwen系列为例, Qwen1.5-MoE-14.3B-A2.7B 的效果与 Qwen1.5-7B 相当接近, 也就是说 MoE 以 2.7B 的激活参数打平 7B 激活参数的 Dense.

--> MoE模型的缺点: 1. 训练不稳定，loss容易急剧上升最终发散  2. 模型更复杂, router/top_k 步骤是离散的，无法反传梯度以训练.需要特定的设计.



如何实现MoE并解决其缺点:
一. routing function 路由网络/方程
总共有 N 个设备, 对于 设备i 上的 L 个tokens: [B, L, D], 每个 token 都有 N 个潜在去处. 如何选择 device-token 之间的映射关系?
主流方案：token 选择 expert --> 对每个设备的每个token, 计算top_k个最该去的expert, 并加权平均k个expert的结果
非主流：expert 选择 token / 共同选择 等, 会存在严重的不可控和遗漏问题（比如有些token没有专家选等），必须引入复杂的兜底机制. 效果也没有 token-choose-expert 好

routing variants(都不常见):
hash routing: 执行一个 D -> 1 的 Hash 映射, 为每个 token 提供确定的 expert 执行 FFN
rl routing: 用强化学习策略的办法, 为每个 token 提供 experts. router nn 输出一些 rl_loss 到 最终的 损失 上
BASE routing: 用线性分配算法的办法，以最小化总成本为目标，求解一个 token-expert 匹配问题


top_k routing: 用一个 D -> k 的小型策略: 为每个 token[,D] 计算 N 个 expert 的 signal, 取 top_k 的 expert 执行 FFN, 最后为每个 token 加权 expert_FFN 作为输出

两个 top_k router variants变种: 主要是在 top_k 的 k 个 expert 择取上做一些变动
1. 专家细粒度拆分 fine-grained segmentation: 本来是 N 个 D->4D->D 的 FFN 中选 top_k, 将它们拆成 2N 个 D->2D->D 的 FFN 中选 top_2k, 此为 fine-grained segment
2. 隔离共享专家 shared expert: 所有tokens都必然择取 shared expert, 以及另外择取 k-1 个 top experts
--> 从 deepseek-MoE ablation 实验报告来看，专家拆分、以及共享专家，都对模型性能有帮助.
--> 从 OlMoE ablation 实验报告来看，专家拆分对模型性能有帮助，而共享专家没有

路由网络 router 如果生成的 signals 是离散选择, 那么会带来训练问题:
attn(token, W_attn) -> act_token
router(act_token, W_rout) -> signal_expert
based on signal_expert, expert_ffn(act_token, W_ffn) -> act_ffn -> .. -> loss

如果 signal_expert 是以mask/indicator等离散的形式参与 act_ffn, 那么 mask/indicator 对 signal_expert = router(act_token, W_rout) 的梯度是0,
导致 loss 无法参与更新 W_rout --> 无法通过loss学会让router选择专家. 不过 W_ffn 和 W_attn 还是是可以得到更新的
另外专家选择缺少负反馈机制 --> 导致大量token趋同性地选择相同的专家, 负载不均.



router的训练问题的解决方法:
原始方法(Gshard from Shazeer 2017): 软路由 soft routing
token_embd 与 N 个代表 expert 的 exprt_embd 分别作 score(相似性) 计算, 截断 top_k (即top_k之外的都下取0) softmax计算得到gating稀疏, 然后将gating稀疏权重作为FFN的权重
这里可以
1.先对scores作softmax计算, 然后再取top_k稀疏 --> deepseek.V1/2
2.先对scores作top_k稀疏(其余都取-inf), 再作softmax计算 --> deepseek.V3
3.在训练时保留所有N个expert的scores并softmax, 作为expert权重, 也就是说在训练时不稀疏以换取极致的可微性. 推理时取top_k experts以换取稀疏性.


主流方法(Switch transformer from Fedus 2022): 辅助均衡损失 balancing loss
不通过主任务(task loss)来update router, 而是构建一个 balancing loss which can lead to update date router 来强行引导 router 均匀分配tokens
这里可以
1. 对于 Batch 内所有 T 个tokens, 可以计算 单个 token x 选择 专家 i 的概率 p_i(x)
2. 将 选择专家i 的概率求平均, 即得到专家i 被选择的概率, 即 P_i = mean(p_i(token)) over all tokens
3. 将 选择专家i 的事实indicator求频率, 即得到专家i 被选择的频率（被top_k选择也一样）, 即 f_i = mean(indicator(token chose expert i)) over all tokens
用 f_i 和 P_i 构建具备负反馈能力的 aux balancing loss, 其中 f_i 作为常数权重无可微参数, P_i 含router参数 可微 可被引导调整router权重以平衡负载
经典的 aux balancing loss = alpha * N * sum_over_N_experts(f_i * P_i)
--> 这个点积具备负反馈能力: 当专家i的实际负载很高时(f_i很大), 梯度下降会迫使router降低P_i, 即降低专家i的评分, 从而推流量给其他专家, 迫使专家分配平衡
--> f_i作为无参常数, P_i由含router参数的softmax产出, 可微
--> 显然 aux balancing loss 是一种间接学习: router并没有学习哪个专家适合哪个token, 而是学如何让所有专家都能处理token. 而专家的处理能力则交给专家网络在主任务中学
    这种“强制平衡”同时解决了“路由器不知道落选的专家是不是更好”的问题：因为负载均衡压力，router会主动尝试把token分配给用得少的专家
